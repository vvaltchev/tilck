================================================================================
  Annotated commit history -- Tilck
================================================================================

This file walks Tilck's first-parent master history in chronological order
(oldest first). Each commit has:

  - its verbatim git log header (hash, author, date, subject, body),
  - the body (when present) as a plain paragraph, and
  - an "AI notes" section summarizing what changed (mechanical, read
    directly off the diff) and, when non-obvious, a context line that
    guesses intent. Every guess is explicitly flagged as such.

Purpose: a pre-processed skim layer over `git log` so that an AI assistant
(or a human) can orient quickly without re-reading every diff. The source
of truth is always `git show <sha>`.

Commit-subject prefix convention: Tilck uses bracketed subsystem prefixes
in subjects -- [boot], [kernel], [mm], [sched], [fs], [vfs], [kb], [tty],
[pci], [acpi], [kmalloc], [signal], [term], [selftest], [gtests],
[toolchain], [build], [ci], [docs], [configurator], [pkgmgr], etc.
Multi-area commits use multiple brackets: [kernel][fs]. When a note says
"touches X subsystem" it is echoing that prefix plus the diff's file
layout.

================================================================================
  ERA 1 -- Cygwin / Bochs on Windows  (Mar 2016 - Aug 2016)
================================================================================

  Early exOS. Development happens on Windows 7/10 under Cygwin, debugged
  in Bochs. The bootloader is hand-written NASM; the kernel is a single
  C file compiled with Cygwin-gcc and linked with the pei-i386 default
  linker script (so symbols carry the underscore prefix). Much of the
  early interrupt / IDT / IRQ scaffolding is adapted from Bran's
  Kernel Development Tutorial, a common starting point for osdev
  hobbyists of the era. By the end of this era Tilck has: a 2-stage
  bootloader, a 32-bit higher-half kernel at 0xC0100000, basic paging
  with map_page / unmap_page / pdir_clone, a physical page bitmap, a
  VGA text-mode console with scrollback, a keyboard driver with
  E0-extended scancodes, and a first ring-3 transition via a synthetic
  iret frame. Everything still lives in per-directory hand-written
  Makefiles; no unit tests, no CI, no .gitignore.

## 342b9ff8 -- finally FULL 32 bit mode works.

Commit: 342b9ff8ccde4dd7107f0af8c345e6e08d2ac651
Author: Vladislav Valtchev <vladi32@gmail.com>
Date:   Sun Mar 6 12:25:36 2016 +0200

AI notes:

What changed:

  boot_stage2.asm reshuffles the protected-mode transition. The old flow (set PE
  -> data-segment reload at complete_flush -> call smart_enable_A20 -> far-jump
  to C) is reordered to: enable A20 while still in real mode, zero the IDTR (no
  IDT yet), set CR0.PE, far-jump 0x08:0x1000 into 32-bit complete_flush, reload
  DS/ES/FS/GS/SS = 0x10, then far-jump 0x08:0x00022000 to the C kernel. Dead
  8-nop delay removed; the speculative [0xb8000] character write is kept as a
  visual heartbeat.

Context (subject says it):

  celebration commit -- the first time 32-bit mode is entered cleanly
  end-to-end.

## 2f8d7035 -- adding an option to avoid zero-init vars in bss

Commit: 2f8d703529...
Author: Vladislav Valtchev <vladi32@gmail.com>
Date:   Mon Mar 14 01:23:32 2016 +0200

AI notes:

What changed:

  - Makefile -- adds -fno-zero-initialized-in-bss to CFLAGS. GCC normally puts
    zero-initialized globals in .bss; this flag forces them into .data instead
    so they get copied along with the rest of the loaded image (the bootloader's
    current objcopy -j .text -j .rdata -j .data flow drops .bss).

  - kernel32.c (+26 / -9) -- likely small cleanups related to globals now safely

Context (subject):

  kernel image doesn't yet allocate and zero .bss at runtime, so any int
  global_x; was reading uninitialized floppy bytes -- this flag works around
  that by turning .bss globals into .data entries with explicit zero
  initialization.

## 0d313234 -- fixing cow (#17)

Commit: 0d31323435...
Author: Vladislav Valtchev <vladi32@gmail.com>
Date:   Sat Dec 17 14:30:13 2016 +0200

Message:

  Fixing the incorrect implementation of COW by using ref-count.
  Now, after fork() both parent and child have COW pages. Anyone who
  writes on a given page, copies it and decreases its ref-count, if
  its ref-count is > 1. In the case its ref-count is 1, that means
  that nobody else refers with COW on that page, so the page is used
  directly after marking as "regular".

AI notes:

What changed:

  - src/i386/paging.c (+110 / -52) -- big rewrite of COW handling:

      * New u16 *pageframes_refcount pointer, allocated at paging init
        to hold a 16-bit refcount per physical page.
      * handle_potential_cow(vaddr) extracted from handle_page_fault:
        reads refcount; if == 1, converts the PTE to a plain writable
        page in place; if > 1, allocates a fresh page, decrements the
        source page's refcount, copies the content, and rewires the
        PTE. Returns true/false so handle_page_fault can distinguish
        COW faults from real ones.
      * pdir_clone() now increments refcount on every shared page
        rather than only-marking-parent-as-COW.

  - src/pageframe_allocator.c (+6 / -1) -- exposes a refcount
    getter/setter pair for paging.c to touch.

  - include/paging.h (+20 / -11), include/commonDefs.h (+3),
    include/process.h (+1 / -1) -- declarations.

  - init_src/init.c (+3 / -1), src/main.c (+5 / -1),
    src/i386/process.c (+9 / -2), src/syscalls.c (+2 / -1) -- callers.

Context (PR body):

  The earlier COW was broken because after the fault the parent's
  page was left read-only even though it no longer needed to be.
  Refcounting lets the last-writer-wins path skip the copy

## d6ac2e2d -- Introducing kernel preemption for user processes (#27)

Commit: d6ac2e2d16...
Author: Vladislav Valtchev <vladi32@gmail.com>
Date:   Sun Apr 30 15:17:54 2017 +0300

Message:

  This change makes each user thread to have its own kernel stack,
  that will be used by the kernel when an interrupt (or syscall)
  occurs. That code runs by default with preemption disabled but it
  can be enabled as well, as happens in the DEMO implementation of
  sys_waitpid(). Therefore, the kernel can actually preempt itself
  and correctly account the ticks that it spent working on a job
  for a user process. In addition to that, a kernel_yield()
  function has been added. That function allows kernel code to save
  its current state and call the scheduler. Overall, all that
  brings exOS much closer to modern operating systems like Linux.

  [plus a 15-bullet commit-by-commit log ending in: implementing
  kernel_yield and making printk disable preemption.]

AI notes:

What changed:

  - kernel/process.c (+205 / -47), kernel/arch/i386/process32.c
    (+50 / -35) -- each task gets its own kernel stack allocated at
    task-create; the TSS is updated before context-switching into a
    task so any ring transition (int, syscall) lands on that task's
    kernel stack instead of a shared one.

  - kernel_yield() -- kernel-side cooperative yield: saves the
    caller's state into the task struct and calls schedule(). Unlike
    a regular schedule after interrupt, this works from arbitrary
    kernel code. Enables code like sys_waitpid() to block without
    holding the CPU.

  - kernel/arch/i386/kernelAsm.asm (+70 / -18) -- asm support for the
    new kernel-stack-per-task model and for kernel_yield.

  - running_in_kernel flag + preemption-enabled / disabled nesting
    counter, maintained across nested interrupts so accounting
    stays correct.

  - Tick accounting: timer_ticks splits into "kernel" vs "user"
    ticks per task, so top-like tools can eventually report CPU
    usage. process.c's counters are reset on task destroy.

  - kernel/main.c (+27 / -26), kernel/arch/i386/paging.c (+3 / -3),
    kernel/include/arch/i386/arch_utils.h (+31 / -17),
    kernel/include/arch/i386/process_int.h (+6 / -3),
    kernel/include/process.h (+15 / -8), kernel/timer.c (+11 / -9),
    usermode_init/init.c (+5) -- matching plumbing. printk() now
    disables preemption around VGA writes to avoid tearing.

Context (PR body):

  Foundation for true preemptive multitasking: a user task can now
  be preempted even while the CPU is running kernel code on its
  behalf. Previously kernel code was effectively non-preemptible,
  making long syscalls (like waitpid) a CPU hog.
